# 介绍
- **处理机**：就是 CPU 或 CPU 核心
- **单处理机系统**：指硬件上只有一个 CPU 或一个核心。
- **多处理机系统**：指硬件上有多个 CPU 或多个核心。这是物理基础，决定了系统能做什么。
- **单道程序环境**：指内存中始终只有一道程序，执行完才换下一道，系统资源利用率很低。
- **多道程序环境**：指内存中同时存放多道程序，当前程序暂停时立刻切换执行另一道，让系统资源得到充分利用。
- **并发**：是逻辑上的同时，在单处理机上靠快速切换让多道程序看起来都在推进，但任一时刻实际只有一道在执行。
- **并行**：是物理上的同时，必须在多处理机上才能实现，同一时刻有多道程序在不同的处理器上真正同时执行。

单处理机只能并发不能并行，操作系统通过多道程序环境实现并发。多处理机才能实现真正并行。


# 并发

并发性是指在一段时间内宏观上有多个程序在同时运行

因为CPU执行速度很快，所以将**按时间片划分，轮流分配给各个程序使用**，故微观上这些程序只能是**分时地交替执行**。

**时间片**：就是操作系统分配给每个正在运行的程序，允许它**一次性连续使用CPU的最长时间**。

> [!tips]-
> 例如，在 1 秒钟时间内，0 - 15 ms 程序 A 运行；15 - 30 ms 程序 B 运行；30 - 45 ms 程序 C 运行；45 - 60 ms 程序 D 运行，因此可以说，在 1 秒钟时间间隔内，宏观上有四道程序在同时运行，但微观上，程序 A、B、C、D 是分时地交替执行的。

# 进程/线程

- 应用程序：放在硬盘上的文件 `WeChat.exe` 叫应用程序/程序。
- 进程：是“活”的，是一次运行活动；只有双击运行程序，程序被加载到内存里开始执行，才产生一个进程。

一个应用程序可以同时被多次加载运行，从而产生多个进程。

> [!NOTES]- 线程
> **我们启动两个进程，一个用来打游戏，一个用来播放音乐**。但是想象一下，如果一个应用程序需要执行的任务非常多，例如 LOL 游戏吧，光是需要播放的音乐就有非常多，人物本身的语音，技能的音效，游戏的背景音乐，塔攻击的声音等等等，还不用说游戏本身，就光播放音乐就需要创建许多许多的进程，而进程本身是一种非常消耗资源的东西，这样的设计显然是不合理的。更何况大多数的操作系统都不需要一个进程访问其他进程的内存空间，**也就是说，进程之间的通信很不方便，此时我们就得引入“线程”这门技术，来解决这个问题。**

#question 偏底层的问题:为什么进程消耗资源多?线程消耗资源小, 

**线程是指进程中的一个执行任务（控制单元），一个进程可以同时并发运行多个线程**。我们可以打开任务管理器，观察到几乎所有的进程都拥有着许多的「线程」(在 WINDOWS 中线程是默认隐藏的，需要在「查看」里面点击「选择列」，有一个线程数的勾选项，找到并勾选就可以了)。

## 区别

**进程**：有独立的内存空间，进程中的数据存放空间（堆空间和栈空间）是独立的，至少有一个线程。

**线程**：堆空间是共享的，栈空间是独立的，线程消耗的资源也比进程小，相互之间可以影响的，又称为轻型进程或进程元。




# 多线程
## 性能

多线程运行程序必然会引入的额外开销

> [!notes]- 线程自身的管理开销
> 这是一个更底层的原因。线程在不同CPU核心间切换，或不同线程修改相邻内存数据，都可能导致CPU缓存里的数据频繁失效，需要重新从较慢的内存中读取，造成显著的性能损失

> [!notes]- 上下文切换开销
> 在单核CPU上，每从一个线程切换到另一个，都需要保存当前线程的状态（如寄存器、程序计数器等），再加载下一个线程的状态，这个过程就是“上下文切换”。切换本身是需要消耗CPU时间的，如果切换过于频繁，CPU就忙于“切换”而非“执行”任务，整体效率自然下降。

> [!notes]- CPU缓存失效率(Cache Thrashing)
> 线程在不同CPU核心间切换，或不同线程修改相邻内存数据，都可能导致CPU缓存里的数据频繁失效，需要重新从较慢的内存中读取，造成显著的性能损失

> [!notes]- 同步与锁竞争开销
> 多线程访问共享资源（如全局变量）时，必须用锁（如互斥锁）来保证数据一致性，防止数据错乱;当一个线程持有锁时，其他需要该锁的线程就会被阻塞、等待;让被阻塞的线程无所事事，还引发了新的上下文切换

但**资源利用率更好**:原理是 I/O 等待时 CPU 空闲，多线程可以让 CPU 趁一个线程等磁盘/网络时切去干别的活。

> [!tips]- 例子
> 单线程串行：5秒读A → 2秒处理A → 5秒读B → 2秒处理B = **14秒**  
多线程交替：5秒读A → 5秒读B（同时处理A，2秒）→ 2秒处理B = **12秒**
等 I/O 的时间被利用起来了。

**程序设计更简单**:单线程要手动管理多个文件的读取和处理状态，代码复杂。多线程直接每个文件起一个线程，一个线程只管一个文件，读的时候被阻塞，其他线程继续跑。逻辑更直，代码更清晰。

**程序响应更快**:一个操作里夹着很多弱关联任务（如发邮件、写日志），单线程得全跑完才能返回给用户。多线程把主线程腾出来先响应用户，那些琐碎活扔给其他线程后台慢慢干，用户不用干等，体验更流畅。


## 随机性

因为一个进程中的多个线程是并发运行的，那么从微观角度上考虑也是有先后顺序的，**那么哪个线程执行完全取决于 CPU 调度**，程序员是控制不了的。  我们可以把多线程并发性看作是多个线程在瞬间抢 CPU 资源，谁抢到资源谁就运行，这也造就了**多线程的随机性**。

> [!tips]- 例子
> 线程A执行1个任务,线程B执行1个任务, 可能会出现两种不同的顺序: 先A后B,先B后A

## 线程安全

当多个线程**同时访问同一个共享资源**时，如果不加控制，就会出现**线程不安全**的问题。

- 多个线程**只读取**同一个公共资源，因为没有写操作，每个线程看到的数据一致，**不会出问题**。
- 多个线程**只写入**同一个资源,就会使实际,与期望的公共资源**数据不一致**;

以下是用c语言实现的例子
```c
int pthread_create(pthread_t *thread,              // 参数1: 用来保存线程被分配的id,
                   const pthread_attr_t *attr,     // 参数2: 设置线程属性,默认就行
                   void *(*start_routine)(void *), // 参数3: 回调函数, 要这个线程执行什么任务
                   void *arg);                     // 参数4: 传递的参数
                   
void* thread_func(void* arg){
	// 任务
}
```

```c
#include <stdio.h>
#include <pthread.h>

#define THREAD_COUNT 5
#define INCREMENTS 100000

int counter = 0;

// 创建线程的回调函数
void* thread_func(void* arg){
    int id = *(int*)arg;
    for(int i = 0; i < INCREMENTS; i++)
        printf("Thread %d: counter is %d\n",id, ++counter);
    return NULL;
}

int main(){
    pthread_t threads[THREAD_COUNT];
    int thread_ids[THREAD_COUNT];
    for(int i = 0; i < THREAD_COUNT; i++){
        thread_ids[i] = i;
        pthread_create(&threads[i], NULL, thread_func, &thread_ids[i]);
    }
    
    for(int i = 0; i < THREAD_COUNT; i++){
        pthread_join(threads[i], NULL);
    }
    
    printf("期望总数: %d\n", THREAD_COUNT * INCREMENTS);
    printf("实际总数: %d\n", counter);
    return 0;
}
```

#竞态条件:多个线程并发访问共享资源，且最终结果依赖于它们执行的相对顺序或时机。上述写入操作就是这种情况

**原因**：在cpu层面, count++并非是**原子化操作**，而是分为三步。 
#原子化操作 -> 操作不可再分(即只用一步完成)
```text
1. load i : 从内存中读取数据
2. i += 1 : 改数据
3. store i : 将数据写回内存当中 
```

> [!tips]- 上述案例的最小化出错情况
> 假设2个线程A,B想要让counter+2,但会出现一种情况: 
> A读(counter = 0) -> B读(counter = 0) -> B写(0->1) -> A写(0->1) 
> 最终结果就是counter = 1 而非2 

# 锁
以小见大,想要实现并发编程, 多线程执行任务一定要保证数据一致性,以及各种约束. 锁的作用就是保证线程安全，通常分为两类：悲观锁、乐观锁

## 悲观锁
(Pessimistic Locking)

悲观锁是一种悲观的并发控制策略，它假设数据总是会发生冲突，因此在整个数据操作过程中，**会对数据进行加锁**，防止其他线程对数据进行修改。

- 核心理念：“先上锁，再操作”，确保操作的独占性
- 适用场景：高冲突环境，即多线程并发访问数据的频率较高
- 代表实现：数据库的行锁、表锁、读锁、写锁；Java中的 synchronized 、 ReentrantLock 等


## 乐观锁 
(Optimistic Locking)

乐观锁是一种乐观的并发控制策略，它假设数据不会发生冲突，因此不会预先加锁，而是在操作完成后，通过一定的机制来检测是否发生了冲突。

核心理念：“先操作，后检测”，通过版本控制或CAS操作实现
适用场景：低冲突环境，即多线程并发修改同一数据的频率较低
代表实现：数据库的版本号机制、时间戳机制；Java中的原子类、CPU层面的CAS原子操作等


## 悲观锁实现
悲观锁的实现依赖于系统提供的锁机制，通过阻塞其他线程来保证数据的一致性

- 阻塞等待机制：当一个线程持有锁时，其他需要该锁的线程会被阻塞，进入等待队列
- 操作系统支持：底层通常依赖操作系统的互斥量（Mutex）、信号量（Semaphore）等机制
- 锁的粒度：可以分为粗粒度锁（如表锁）和细粒度锁（如行锁）


## 乐观锁实现
乐观锁的实现主要通过以下两种机制：

> [!notes]- 版本号机制
> 在数据中增加一个版本号（version）字段
> 读取数据时，同时读取版本号
> 更新数据时，比较版本号，如果相同则更新并将版本号加1，如果不同则说明数据已被其他线程修改

>[!notes]- CAS操作
> CAS(Compare-And-Swap) 是一种在软件层面的无锁算法(系统层面还是加锁了)，包含三个操作数：**内存位置（V）**,**预期值（A）**,**新值（B）**
> 只有当内存位置的值等于预期值时，才会将该位置的值更新为新值，否则不执行任何操作。**整个操作在CPU上是原子的。**


## 详细对比分析	

| 特性    | 悲观锁                    | 乐观锁                  |
| ----- | ---------------------- | -------------------- |
| 并发假设  | 总是假设会发生冲突              | 假设不会发生冲突             |
| 实现方式  | 基于锁机制，直接阻塞其他线程         | 基于版本控制或CAS操作         |
| 性能影响  | 有锁竞争时，会导致线程阻塞，上下文切换开销大 | 无锁竞争，无需线程切换，但可能有重试开销 |
| 死锁风险  | 有死锁风险，需要考虑锁的获取顺序       | 无死锁风险                |
| 适用场景  | 高并发写操作、数据冲突频繁          | 读多写少、数据冲突较少          |
| 实现复杂度 | 相对简单，依赖系统锁机制           | 相对复杂，需要自己实现冲突检测逻辑    |
