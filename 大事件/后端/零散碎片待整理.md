# Lombok 注解包
Lombok 是一个优秀的 Java 库，**它通过注解在编译期自动生成样板代码**（如 getter、setter、构造函数等），能显著减少代码冗余，提升开发效率。

查看target目录下能发现编译之后的代码带有getter和setter等代码。
以下是 Lombok 中最常用的一些注解，按功能分类介绍：

1. `@Data`：组合注解，包含 `@Getter`、`@Setter`、`@ToString`、`@EqualsAndHashCode`、`@RequiredArgsConstructor`
2. `@Getter` / `@Setter`：生成 getter / setter 方法
3. `@ToString`：生成 `toString()` 方法
4. `@EqualsAndHashCode`：生成 `equals()` 和 `hashCode()` 方法
5. `@NoArgsConstructor`：生成无参构造器
6. `@AllArgsConstructor`：生成全参构造器

# Validation 注解包
@Validated ：方法上参数上的限制->要在启用检测的类上加这个注解；类上字段限制->传对象参数的时候，要在形参上加；

**@Pattern(regexp = "^\\S{5,16}$")**

- `@Pattern`：Java Bean Validation API中的注解，用于验证字符串是否符合特定的正则表达式
- `regexp`：指定要匹配的正则表达式
- `^`：匹配字符串的开始
- `\\S`：匹配任何非空白字符（反斜杠是Java字符串中的转义字符，实际正则是 `\S`）
- `{5,16}`：前面的字符重复5到16次
- `$`：匹配字符串的结束

# StringUtils包

👇配置好Controller类上的Validation后还要去处理全局异常，因为在传参的时候就会检测，一定要处理！以此返回对应接口文档的响应规范

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        e.printStackTrace();
        return Result.error(StringUtils.hasLength(e.getMessage()) ? e.getMessage() : "操作失败");
    }
}
```



# JWT令牌 java-jwt
Json Web Token

防止不登陆访问资源的行为，访问必须携带令牌(token)

**Token是根据用户名 + 密码(自定义字段名) + 密钥字符 + 算法 + 实效性构成**

失败的三种情况：密钥错误、token被篡改、时间过期

springboot没有自带，需要引入坐标
```xml
    <dependency>
        <groupId>com.auth0</groupId>
        <artifactId>java-jwt</artifactId>
        <version>4.4.0</version>
    </dependency>
```

生成token
```java
    @Test
    public void genToken() {
        Map<String, Object> claims = new HashMap<>();
        claims.put("id", "1");
        claims.put("username", "张三");

        String token = JWT.create()
                .withClaim("user", claims)      // 将整个 claims 对象作为 "user" 字段嵌套
                .withExpiresAt(new java.util.Date(System.currentTimeMillis() + 1000L * 60 * 60 * 3)) // 3 小时后过期
                .sign(Algorithm.HMAC256("palmer"));  // 使用 HMAC-SHA256 签名，密钥为 "palmer"

        System.out.println(token);
    }
```

验证token
```java
    @Test
    public void testParse() {
        String token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9" +
                ".eyJ1c2VyIjp7ImlkIjoiMSIsInVzZXJuYW1lIjoi5byg" +
                "5LiJIn0sImV4cCI6MTc3NzQwODAzN30.o29h1mjSrdxLB2nHzlJk4GanN9Lzhu2uH1xMMmGBoRM";

        JWTVerifier jwtVerifier = JWT.require(Algorithm.HMAC256("palmer")).build();
        DecodedJWT decodedJWT = jwtVerifier.verify(token);
        Map<String, Claim> claims = decodedJWT.getClaims();
        System.out.println(claims.get("user"));
    }
```





# 拦截器 
Interceptors包 WebConfig类

Java Spring Interceptor（拦截器）的主要作用是在请求到达控制器（Controller）前后执行一些通用的逻辑。

Spring Interceptor的核心作用是通过实现HandlerInterceptor接口，在请求的不同阶段插入通用逻辑：
预处理：请求到达控制器前执行，如身份验证、权限检查、日志记录。

后处理：控制器处理完成后，视图渲染前执行，如处理模型数据。

完成处理：整个请求结束后执行，如资源清理、性能监控。
它用于处理跨多个控制器的通用需求（“横切关注点”），减少代码重复，提升维护性。

# 跨域问题

把前端和后端在localhost上一起启用，前端读取不了后端返回的响应数据。
这是跨域问题，基于**浏览器**的同源策略，防止恶意网站跨域攻击。服务器之间没有跨域限制，**只是浏览器限制了JS脚本的跨域请求**发送ajax请求会失败。


**核心机制：**
- 同源策略：浏览器只允许 JS 读取**协议、域名、端口**完全相同的源的响应
- 跨域触发：三者任一不同即为跨域，浏览器会拦截响应数据（但请求通常已发出）
- 安全目的：防止恶意网站冒充你窃取其他网站的数据（如利用Cookie自动附带）
    
**主流解决方案：**
- 开发代理（Vite/Webpack Dev Server 转发 `/api`，最常用）
- CORS（服务器设置 `Access-Control-Allow-Origin` 响应头）服务端在响应头里明确告诉浏览器"我允许某个源来访问"
- Nginx反向代理（生产环境，对浏览器同源）


问题总结
1. CORS 跨域问题
现象： 前端 http://localhost:5173 访问后端 http://127.0.0.1:8080 被浏览器阻止
原因：
Spring Boot 的 CORS 配置使用了 allowedOriginPatterns("*") 与 allowCredentials(true) 不兼容
拦截器拦截了 OPTIONS 预检请求，导致返回 401，CORS 头信息无法正确设置
解决方案：
修改 WebConfig.java：将 allowedOriginPatterns("*") 改为明确指定源 allowedOrigins("http://localhost:5173", "http://127.0.0.1:5173")
在 LoginInterceptor.java 中添加 OPTIONS 请求放行逻辑
在 UserController.java 添加 @CrossOrigin 注解
2. Vue 模板数据绑定问题
现象： 请求成功但页面不显示数据
原因： Vue template 中直接使用 user.username 而不是响应式绑定语法 {{ user.username }}
解决方案：
使用双花括号 {{ }} 进行数据绑定
注意后端返回的数据结构：result.data.data（axios 响应对象 + Result 包装）

核心教训：
CORS 配置时，allowCredentials(true) 不能配合通配符 * 使用，必须明确指定源
拦截器必须放行 OPTIONS 预检请求，否则 CORS 无法正常工作
Vue 3 template 中必须使用 {{ }} 语法来显示响应式数据
